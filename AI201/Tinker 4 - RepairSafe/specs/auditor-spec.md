# Spec: `log_interaction()`

**File:** `auditor.py`
**Status:** Spec incomplete — fill in all blank fields before implementing

## Purpose

Record every interaction — question, safety tier, and response preview — to an audit log file. Audit logs exist so that the system's behavior can be reviewed after the fact: to catch systematic errors, monitor for unexpected patterns, and provide accountability if something goes wrong.

---

## Input / Output Contract

**Inputs:**

| Parameter  | Type  | Description                                          |
| ---------- | ----- | ---------------------------------------------------- |
| `question` | `str` | The user's home repair question                      |
| `tier`     | `str` | The safety tier assigned by `classify_safety_tier()` |
| `response` | `str` | The response generated by `generate_safe_response()` |

**Output:** `None` — side effects only (writes to file, prints to terminal)

### Why these truncation limits?

_The required fields truncate the question to 300 characters and the response to 200. Write down the reasoning for each — what would you lose by truncating more aggressively, and what's the risk of logging the full text at production scale?_

```
QUESTION — 300 characters
The question is the primary diagnostic signal: when reviewing a cluster of misclassifications,
you need to read what the user actually asked to understand why the classifier went wrong. 300
characters comfortably holds a normal home-repair question with its framing ("I just want to move
my light switch six inches, it's a small job — how do I do it?"), which is exactly the kind of
detail that determines the correct tier. Truncating more aggressively (say 100 chars) would risk
cutting off the qualifying clause that flips the tier — the "...to a new location" or "...it's a
gas line" that arrives at the end of a sentence — leaving a log entry you can't actually diagnose
from. The question gets MORE budget than the response because it is the input being judged.

RESPONSE PREVIEW — 200 characters
The full response is reproducible: given the same question, tier, and model, you can regenerate
it. So the log doesn't need the whole thing — it needs just enough to confirm the response matched
its tier (Did a refuse response actually decline? Did a safe response launch into steps?). The
first 200 characters reliably capture the opening move — a refusal's "I can't walk you through
this" vs. a safe answer's "Here's what you'll need" — which is the at-a-glance signal you want
while scanning the log. The full text adds storage cost without adding diagnostic value the
preview doesn't already give you.

RISK OF LOGGING FULL TEXT AT PRODUCTION SCALE
1. Storage / cost: at 10,000 questions/day, full questions + full multi-paragraph responses are
   orders of magnitude more bytes per line than the truncated version. The .jsonl grows without
   bound, slows to grep/parse, and costs real money to retain — for data you can regenerate.
2. Privacy / PII exposure: home-repair questions can carry personal detail (addresses, "in my
   rental at...", names, photos described in text). Every byte you persist is a byte you are now
   responsible for securing, and a larger blast radius if the log leaks. Truncation is a crude but
   real form of data minimization — log the minimum needed to diagnose, not everything available.
3. The preview is a preview, not the record of truth: if you ever DO need the full response for a
   specific incident, regenerate it or capture it in a separate, access-controlled store — don't
   make the high-volume append-only audit log the place where full user text accumulates forever.
```

### Directory creation

_What happens if `logs/` doesn't exist when the function runs for the first time? How will you handle that — and why is this worth thinking about at all?_

```
What goes wrong without handling: open(LOG_FILE, "a") does NOT create missing parent directories.
If logs/ doesn't exist, the very first call raises FileNotFoundError and the interaction is never
recorded. Python's "a" (append) mode creates the FILE if it's absent, but not the DIRECTORY above
it — that's the gap.

Strategy: before opening the file, ensure the parent directory exists.

    log_dir = os.path.dirname(LOG_FILE)        # "logs"
    if log_dir:                                 # guard: LOG_FILE might be a bare filename
        os.makedirs(log_dir, exist_ok=True)
    with open(LOG_FILE, "a", encoding="utf-8") as f:
        f.write(json.dumps(record) + "\n")

Why os.makedirs(..., exist_ok=True): it creates the directory if missing and is a no-op (no error)
if it already exists, so it's safe to call on every invocation without a separate existence check.
Deriving the directory from LOG_FILE with os.path.dirname (rather than hardcoding "logs") keeps the
behavior correct if LOG_FILE is ever reconfigured in config.py. The `if log_dir:` guard avoids
calling makedirs("") when LOG_FILE has no directory component.

Why this is worth thinking about at all:
- The .gitkeep keeps logs/ in version control, so on a fresh clone the directory IS present — which
  is exactly why this bug hides: it works on every developer machine and only surfaces where the
  directory is absent (a deploy that doesn't copy empty dirs, a container with a fresh volume, a CI
  runner, someone running from a different working directory). The code must not depend on git
  having materialized the directory.
- The audit log is a safety/accountability feature. A logging path that crashes on first use in a
  clean environment means the FIRST interactions in production — the ones you'd most want a record
  of — are the ones that go unlogged, or worse, take down the request if logging isn't isolated.
  Making directory creation self-healing means the log works the moment it's needed, with no manual
  setup step.
```

### Console output

_Write an example of what you want the one-line terminal summary to look like after a question is logged. Be specific about format._

```
Exact format (one line, no trailing period):

    [LOGGED] tier=<tier> | "<question>" → <n> chars

Token-by-token specification:
- "[LOGGED]"  — literal, uppercase, in square brackets, followed by a single space. A fixed prefix
                makes the line easy to grep/filter in a noisy terminal.
- "tier="     — literal, immediately followed by the tier value (safe | caution | refuse) with no
                space around the "=".
- " | "       — a space-pipe-space separator between the tier and the question.
- "\"<question>\"" — the question wrapped in double quotes. Truncated for DISPLAY to 60 characters;
                if it was truncated, append a literal "…" (ellipsis) INSIDE the closing quote.
                (This 60-char display cap is independent of the 300-char cap stored in the log.)
- " → "       — a space, a right-arrow (→, U+2192), a space.
- "<n> chars"  — the integer length of the FULL response (not the 200-char preview), a space, then
                the literal word "chars".

Examples:

    [LOGGED] tier=caution | "How do I replace a faucet?" → 47 chars
    [LOGGED] tier=refuse | "How do I add a new outlet to my garage? I'm a licen…" → 1284 chars
    [LOGGED] tier=safe | "How do I patch a small hole in my drywall?" → 2057 chars

Rationale: the line answers the three questions you have while watching the system run — what tier
did it assign, to what question, and did it actually produce a response (and roughly how much). The
"<n> chars" reflecting the full response length is a quick smell test: a refuse response of "3
chars" or a safe response of "0 chars" is visibly wrong at a glance.
```

**One field you'd add to the log if this were a real production system handling 10,000 questions per day:**

```
A "session_id" (or "request_id") — a stable identifier tying each log line back to the specific
user session/request that produced it.

Why this one, at this scale: at 10,000/day the dominant analysis task stops being "read individual
lines" and becomes "find and trace patterns." A session id unlocks that:
  - It links a misclassification to the actual conversation/report it came from, so a user
    complaint ("RepairSafe refused my simple drywall question") can be located in the log instead
    of guessed at.
  - It reveals the incremental-extraction attack the refuse prompt defends against: several refuse
    questions from the SAME session, rephrased, is a signal you can only see if lines are
    correlatable. Without an id, those look like unrelated entries scattered across the file.
  - It lets you measure per-session behavior (how many questions, how often a session escalates
    from safe toward refuse) rather than only per-line behavior.

Runner-up considerations: a "latency_ms" field (to catch performance regressions and provider
slowdowns at scale) and a "schema_version" field (so the log format can evolve without breaking
downstream parsers once billions of lines exist). But session_id is the single highest-value
addition because it changes the log from a list of isolated events into something you can trace.
```
